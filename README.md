# pic18-c-compiler

A tiny **teaching-quality C compiler** for the Microchip **PIC18** family
(default chip: `PIC18F4520`). Built with [ANTLR4][antlr] for parsing and a
hand-written code generator that emits MPASM-style PIC18 assembly.

This is **not** a production compiler. The point is to be small, readable,
and easy to extend in a classroom setting.

The parser uses the upstream **C11 grammar** from [`antlr/grammars-v4`][gv4]
(`CLexer.g4` + `CParser.g4`), so you can write C source the way C source
actually looks. The codegen, however, only handles the small subset
described in [Supported C subset](#supported-c-subset) below — anything else
the grammar accepts will parse fine but be rejected with a clear
"…not supported in this teaching compiler" error from the codegen.

[antlr]: https://www.antlr.org/
[gv4]: https://github.com/antlr/grammars-v4/tree/master/c

## Build, test, run

Requires **JDK 17+** and **Maven 3.9+**.

```bash
# Compile the project (also generates ANTLR parser sources):
mvn -DskipTests package

# Run the unit tests:
mvn test

# Compile examples/blink.c -> examples/blink.asm using the runnable jar:
java -jar target/pic18-c-compiler-0.1.0-SNAPSHOT.jar examples/blink.c -o examples/blink.asm

# Or via Maven (no need to package first):
mvn -DskipTests exec:java \
    -Dexec.mainClass=com.example.pic18.Main \
    -Dexec.args="examples/blink.c -o examples/blink.asm"

# Write the assembly to stdout instead of a file:
java -jar target/pic18-c-compiler-0.1.0-SNAPSHOT.jar examples/blink.c -o -
```

## Grammar

`CLexer.g4` and `CParser.g4` under `src/main/antlr4/com/example/pic18/grammar/`
are the **unmodified C11 grammars** from
[antlr/grammars-v4](https://github.com/antlr/grammars-v4/tree/master/c).

Both grammars declare a `superClass` (`CLexerBase` / `CParserBase`) for
runtime helpers — semantic predicates that disambiguate things like
typedef-names from identifiers, and action methods that maintain a parse-time
symbol table. The upstream Java implementations of those base classes are
big enough to dwarf the rest of this project — `CLexerBase` shells out to a
system `gcc`/`clang` to preprocess the input, and `CParserBase` carries a
~600-line C symbol table. Both are wildly out of scope for a teaching
compiler that doesn't even support `typedef` or `#include`.

We therefore ship **minimal stub** versions of those base classes under
`src/main/java/com/example/pic18/grammar/`:

* `CLexerBase` is a no-op pass-through. The grammar's existing
  `MultiLineMacro` / `Directive` / `LineDirective` lexer rules already route
  any stray `#…` lines to a hidden channel, so the lexer just ignores them
  rather than running a preprocessor.
* `CParserBase` implements every semantic predicate
  (`IsStatement`, `IsDeclaration`, `IsCast`, `IsTypedefName`, …) with a
  small token-lookahead heuristic, and gives every action method
  (`OutputSymbolTable`, `EnterScope`, …) an empty body.

Because we don't track typedefs, source like `typedef int T; T x;` will not
parse the second statement as a declaration. That's fine: the codegen
rejects `typedef` up front anyway. Everything in the Supported C subset
below works.

The parser is therefore the *only* layer that "understands C11". Anything
the grammar accepts but the codegen doesn't (pointers, structs, casts,
sizeof, switch, do-while, goto, string literals, multiple declarators per
declaration, compound assignment, etc.) is reported as a clear
"…not supported in this teaching compiler" error from the codegen rather
than as a parse error.

## Supported C subset

| Feature                              | Status                                          |
| ------------------------------------ | ----------------------------------------------- |
| Primitive types                      | `int`, `char`, `void` (and `unsigned` aliases). All integer types are **8-bit** on this target. |
| Global variables                     | With optional **constant** initializer.         |
| Function definitions                 | Including `int main(void)` / `void main(void)`. |
| Function prototypes                  | `void foo(int n);` (forward declaration).       |
| Local variables                      | Per-function statically allocated bytes (no real stack). |
| Statements                           | `{}` blocks, `if`/`else`, `while`, `for`, `return`, expression statement, assignment. |
| Arithmetic                           | `+`, `-`, `*` (low byte of `MULWF`).            |
| Division / modulo                    | Only by **constant power-of-two** (lowered to `RRCF`/`ANDLW`). Other divisors are a compile error. |
| Bitwise / shift                      | `&`, `\|`, `^`, `~`, `<<`, `>>`. Shift count must be a constant integer literal. |
| Comparison                           | `== != < <= > >=`, lowered to `CPFSEQ`/`CPFSGT`/`CPFSLT`. |
| Logical                              | `&&`, `\|\|` with short-circuit lowering.       |
| Unary                                | `!`, `-`, `~`.                                  |
| Increment / decrement                | `v++`, `v--`, `++v`, `--v` on a bare identifier (local or global). Lowered to in-place `INCF`/`DECF`. |
| Literals                             | Decimal `42`, hex `0x2A`, char `'A'` (and `\n \r \t \\ \' \" \0`). |
| Built-in functions                   | `void out(int port, int value);`<br>`int in(int port);`<br>`void delay(int n);` |

### Built-in helpers

These are **ordinary identifiers** (not reserved words) that the codegen
recognises specially:

* `out(port, value)` — writes `value` (an arbitrary expression) to `LATA`,
  `LATB`, `LATC`, `LATD` or `LATE` for `port` 0..4. The `port` argument **must
  be a compile-time integer constant** in `0..4`.
* `in(port)` — reads `PORTA`..`PORTE`. Same `port` constraint.
* `delay(n)` — calls a small `F_delay` runtime helper that does a nested
  decrement loop (outer = `n`, inner = `0xFF`). The `n` argument may be any
  expression.

You may (and should) declare prototypes for these in your C source so the
compiler accepts the source as idiomatic C:

```c
void out(int port, int value);
int  in(int port);
void delay(int n);
```

## Generated assembly: shape

The compiler emits MPASM-style assembly for the **PIC18F4520** with the
following overall structure (see `examples/blink.asm` for a complete example):

```asm
        LIST    P=18F4520
        #include <p18f4520.inc>
        CONFIG  OSC = INTIO67   ; tweak per your hardware
        CONFIG  WDT = OFF
        ...

ACS_VARS UDATA_ACS              ; access-bank variables
R0..R7                  RES 1   ; scratch registers
ARG0..ARG3              RES 1   ; calling-convention arg slots
RETVAL                  RES 1   ; return value mirror
DELAY_OUTER/INNER       RES 1   ; delay loop counters
EVAL_STACK              RES 16  ; expression evaluation stack
G_<global>              RES 1   ; one byte per global
F_<fn>__LOC_<v>         RES 1   ; one byte per function-local

RESETV  CODE  0x0000            ; reset vector
        GOTO    MAIN
HIISR   CODE  0x0008            ; hi-priority ISR placeholder
        RETFIE  FAST
LOISR   CODE  0x0018            ; lo-priority ISR placeholder
        RETFIE  FAST
PROG    CODE                    ; program code
MAIN:   ...                     ; user functions
F_<fn>: ...
F_delay:...                     ; runtime helpers (only if used)
        END
```

### Calling convention

* All values are 8-bit.
* Up to **4 byte-sized arguments** are passed in fixed RAM slots
  `ARG0..ARG3`. The caller stores them just before `CALL`.
* Return value lives in `WREG` and is also mirrored to `RETVAL` so callers
  can read it after `CALL` (we copy `RETVAL` → `W` after the call site).
* Calls use the **PIC18 hardware return stack** (31 entries). Recursion
  works in principle but **locals are static** — recursive calls clobber
  each other's locals.

### Expression evaluation

Every expression evaluates to a byte in `WREG`. Binary operators compile the
left operand to `W`, push it to the next slot of `EVAL_STACK`, evaluate the
right operand to `W`, then perform the actual ALU op against
`EVAL_STACK+(sp-1)`. The compiler tracks `sp` at compile time and reports
the maximum depth at the end of each function as a comment, so you can size
`EVAL_STACK` appropriately. Default capacity: 16 slots.

## Limitations

This is a teaching project; many real-C things are deliberately missing or
simplified:

* **8-bit `int`.** `int`, `char`, and `unsigned int` are all a single byte.
  Larger values truncate silently (with a warning comment).
* **No real stack for locals.** Each function's locals are statically
  allocated bytes named `F_<fn>__LOC_<var>`. A recursive call into the
  same function therefore aliases its parent's locals. Recursion of
  *parameter-less, local-less* functions is fine.
* **Maximum 4 arguments per function**, all byte-sized. (Increase
  `MAX_ARGS` and add more `ARG*` slots if you need more.)
* **No `struct`, `union`, `typedef`, `enum`, arrays, pointers, casts,
  `sizeof`, or strings.** The language really is just integers and control
  flow. The C11 grammar parses all of those, but the codegen rejects them
  with `"<feature> not supported"` errors.
* **No `switch`, `do/while`, `goto`, `break`, `continue`, labels, ternary
  `?:`, or compound assignment (`+=`, `-=`, …).** Same story — parsed,
  rejected by the codegen.
* **No real preprocessor**: `#include`, `#define`, `#ifdef` are not
  expanded. The C11 lexer routes any `#`-prefixed lines to a hidden channel
  (the stub `CLexerBase` is a pass-through, not a real preprocessor), so
  they're silently ignored rather than expanded. If you need them, run the
  source through `cpp` first.
* **No floating point.**
* **Division & modulus by a non-power-of-two is a compile error.** Constant
  power-of-two divisors are lowered to repeated `RRCF`/`ANDLW`. A real
  software-divide runtime is left as an exercise.
* **Variable shifts are unsupported.** The shift count must be a compile-time
  integer literal.
* **Block scoping is flattened.** All locals declared anywhere inside a
  function share one flat scope at the function level. Re-declaring a name
  in an inner block is a compile error.
* **No TRIS configuration.** `out()` writes `LAT*`, but doesn't configure the
  pin direction. You must edit the generated assembly to set `TRIS*` bits
  (or add a small init stub) for outputs to actually drive pins.
* **Initialised globals don't get initialised.** They're allocated in
  uninitialised RAM (`UDATA_ACS`). On a real PIC18, you'd need C-runtime-style
  init code to copy initial values in. We just store the requested initial
  value in a comment and leave the variable zeroed at reset by the chip.
* **The generated assembly is targeted at MPASM syntax** (`P=18F4520`,
  `UDATA_ACS`, `CODE`, `RES`, `CONFIG`, `, A` access-bank specifier). It
  may need tweaks to assemble cleanly with other PIC18 toolchains. The
  intent is that it would assemble in MPLAB IDE / `mpasmx` for the
  supported subset, but we don't run the assembler in CI.

### Next steps (good extension exercises)

* Implement a real software-divide runtime helper.
* Promote `int` to 16-bit with a paired-byte representation.
* Add proper TRIS / OSCCON setup as a `setup()` prelude option.
* Add basic-block based local register allocation instead of the
  always-spill stack-machine model.
* Add `#include` / very simple preprocessor support.
* Real block scoping & shadowing.

## Example: `examples/blink.c`

```c
void delay(int n);
void out(int port, int value);

int main(void) {
    int i;
    out(1, 0x00);              // LATB <- 0
    while (1) {
        out(1, 0x01);          // LATB <- 0x01 (RB0 high)
        delay(200);
        out(1, 0x00);          // LATB <- 0x00 (RB0 low)
        delay(200);
    }
    return 0;
}
```

Compile it:

```bash
mvn -DskipTests package
java -jar target/pic18-c-compiler-0.1.0-SNAPSHOT.jar examples/blink.c -o examples/blink.asm
```

The first ~30 lines of the generated assembly look like:

```asm
        LIST    P=18F4520
        #include <p18f4520.inc>
        CONFIG  OSC = INTIO67
        CONFIG  WDT = OFF
        ...
ACS_VARS UDATA_ACS
R0                      RES     1
...
EVAL_STACK              RES     16

RESETV  CODE    0x0000
        GOTO    MAIN
...

MAIN:
        CLRF    F_main__LOC_i, A
        ; out(1 /* LATB */, expr)
        MOVLW   0x00
        MOVWF   LATB, A
        ; while (...)
WHILE_0:
        MOVLW   0x01
        IORLW   0
        BZ      WEND_1
        ; out(1 /* LATB */, expr)
        MOVLW   0x01
        MOVWF   LATB, A
        ; delay(n)
        MOVLW   0xC8
        MOVWF   ARG0, A
        CALL    F_delay, 0
        ...
```
